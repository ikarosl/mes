# 数据库设计问题与解决方案交接文档

## 1. 文档目的与当前状态

本文用于记录生产业务完整工作流与 `docs/new.md` 核查过程中已经确认的问题、最终决策、后续数据库设计修改点、应用事务规则和验收要求，作为更换编码场景后的暂停交接材料。

本次只新增交接文档，不修改数据库设计、业务代码或迁移。后续恢复工作时，应先依据本文修订 `docs/new.md` 和 `docs/业务完整工作流.md`，完成一致性核查后，再进入对应业务模块的迁移与实现。

当前状态与不可突破的边界如下：

- `docs/new.md` 仍是数据库业务设计的唯一基准；本文是待落实的审查结论，不能替代 `docs/new.md`。
- 当前只完成 RBAC、认证、操作日志和前端权限基础设施迁移。
- 产品、工艺、生产、库存、质量和追溯业务表尚未迁移，相关业务代码不得提前迁入。
- 数据库变更只能在 `packages/database/migrations` 追加迁移；已执行迁移不得修改。
- 库存数量只以 `inventory_transaction` 为事实来源，任何主表、批次表和汇总视图不得成为第二库存事实来源。
- 生产需求只以 `production_item_demand` 为事实来源，汇总视图不得被写入。
- 项目继续采用轻量 MES 模块化单体，不引入完整 ERP/MES 范围，也不拆分微服务。
- 所有核心确认动作必须在同一事务内写入业务事实、库存事实、状态变化和操作日志；日志不得记录密码、Token、Cookie 或其他密钥。

## 2. 已确认问题与最终解决方案

### 2.1 项目数据隔离

#### 现状与问题

当前设计没有 `projects` 表，也没有业务表的 `project_id`，因此不能实现不同项目之间的数据隔离。如果界面仍使用“进入项目”的表达，容易让使用者误以为系统已经提供项目级权限、主数据和库存隔离。

#### 最终决策

当前轻量 MES 不引入项目模型，不扩展完整 ERP/MES 范围。“进入项目”应理解或改写为“进入生产管理”或“进入业务模块”。

#### 数据库与应用处理

- 不创建 `projects`、`project_members`。
- 不给工单、生产批次、库存单据或质量记录增加 `project_id`。
- 前端不得展示系统尚未具备的项目隔离能力。
- 只有未来明确提出多项目隔离需求后，才单独评审产品、工艺、库存批次等主数据是全局共享还是项目私有；在该决策完成前不得局部增加 `project_id`。

#### 验收要求

- 数据库设计和业务工作流不得宣称支持项目级隔离。
- 本次后续修订不得出现新的 `projects`、`project_id` 或项目权限模型。

### 2.2 外部订单号

#### 现状与问题

`work_orders` 没有结构化外部订单号，当前只能暂存在 `remark`，无法稳定查询或建立上游订单与生产工单之间的绑定。

#### 最终决策

只给 `work_orders` 增加一个可空外部订单号和普通索引：

```text
external_order_no VARCHAR(100) NULL
INDEX (external_order_no)
```

#### 数据库与应用处理

- 不增加 `external_order_line_no`、`source_system`。
- 不给 `external_order_no` 建立唯一约束，允许同一个外部订单拆分为多个生产工单。
- 新业务不得继续把结构化外部订单号写入 `remark`。
- 外部订单号不作为 MES 工单自身的唯一标识；`work_order_no` 仍保持唯一。

#### 验收要求

- 可以按外部订单号查询零个、一个或多个工单。
- 同一外部订单号创建多个工单时数据库不得报唯一键冲突。
- 空外部订单号不影响内部生产工单创建。

### 2.3 产品与工艺路线归属

#### 现状与问题

普通外键只能保证 `products.default_route_id` 和 `production_batches.route_id` 指向一条存在的工艺路线，不能保证该路线属于相同产品。前端自动带出默认路线只能改善交互，不能作为数据一致性边界。

#### 最终决策

采用轻量应用事务校验：不增加复杂组合外键，也不在 `batch_step_records` 冗余 `route_id`。

#### 数据库与应用处理

- 设置产品默认路线时，应用必须校验路线的 `product_id` 等于当前产品 ID，且路线状态为 `enabled`。
- 创建生产批次时，以工单产品为准；未指定路线时读取产品默认路线，指定路线时允许使用同产品的非默认路线。
- 生产批次不得使用其他产品的路线或未启用路线。
- 批次工序必须由后端查询所选路线的有效 `process_route_steps` 后按顺序自动生成。
- 后端不接受前端提交任意 `route_step_id` 集合作为批次工序来源。
- 上述读取、校验、批次创建和批次工序生成必须处于同一应用事务。
- `docs/new.md` 应明确标注：路线归属属于 application 事务不变量，前端默认填充不构成完整性保障。

#### 测试要求

- 产品 A 不能设置产品 B 的路线为默认路线。
- 产品 A 的生产批次不能选择产品 B 的路线。
- 生产批次允许选择产品 A 的非默认启用路线。
- 停用或归档路线不能用于创建新生产批次。
- 篡改前端请求中的路线步骤列表不能生成不属于批次路线的工序记录。

### 2.4 工序规则快照与跳过状态

#### 现状与问题

`process_route_steps` 已保存 `need_record` 和 `need_inspection`，但 `batch_step_records` 没有冻结批次创建时的规则。历史批次完成条件如果直接读取后来变化的路线配置，可能发生漂移。同时，当前工作流没有正式的工序跳过配置，却在状态字典中出现了 `skipped`。

#### 最终决策

批次工序冻结必须报工、必须检验规则；当前业务不支持跳过工序。

#### 数据库与应用处理

给 `batch_step_records` 增加：

```text
need_record_snapshot     TINYINT NOT NULL
need_inspection_snapshot TINYINT NOT NULL
```

并执行以下调整：

- 从 `batch_step_records.status` 允许值中删除 `skipped`。
- 从业务工作流中删除“不允许跳过的工序”及异常状态示例中的 `skipped`。
- 不增加 `allow_skip`、`allow_skip_snapshot`。
- 创建生产批次工序时，从路线步骤复制两个规则快照；后续路线调整不得回写历史批次工序。
- 批次完工条件只读取批次工序快照，不读取当前路线配置。

#### 测试要求

- 批次创建后修改尚未启用的新路线版本，不影响既有批次工序规则。
- 必须报工工序未完成时批次不能完工。
- 必须检验工序没有有效检验结论时批次不能完工。
- 任何接口都不能把工序状态写为 `skipped`。

### 2.5 生产需求生成幂等

#### 现状与问题

重复点击“生成需求”、客户端超时重试或服务端重复消费可能为同一生产批次和 BOM 行重复创建需求，导致缺料、分配和领料结果被放大。

#### 最终决策

`production_item_demand` 增加唯一幂等键：

```text
idempotency_key VARCHAR(150) NOT NULL
UNIQUE (idempotency_key)
```

#### 键生成规则

```text
正常需求：NORMAL:{production_batch_id}:{product_material_id}
报废补料：SCRAP:{source_scrap_id}:{product_material_id}
人工追加：ADDITIONAL:{production_batch_id}:{business_action_no}:{product_material_id}
```

其中 `business_action_no` 必须是一次人工追加动作的稳定唯一编号，不能使用随机重试值。

#### 应用事务规则

- 同一个幂等键重复提交时返回既有结果，不再插入新需求。
- 需求事实和对应操作日志必须在同一事务写入。
- 幂等冲突不得通过修改既有 `need_number` 绕过。
- 正常需求生成后，BOM 或批次计划数量变化不得自动覆盖既有需求。

#### 测试要求

- 同一正常需求命令连续执行两次只产生一条需求。
- 两个并发相同请求只成功插入一条需求。
- 同一次报废针对不同 BOM 行可以分别生成补料需求。
- 相同报废和相同 BOM 行的重试不会重复生成补料需求。

### 2.6 一次报废生成多种补料需求

#### 现状与问题

`production_item_demand` 当前的 `UNIQUE (source_scrap_id)` 把报废与补料需求限制为一对一，无法表达一件在制品报废后需要同时补充多种 BOM 物料。

#### 最终决策与数据库修改

- 删除 `UNIQUE (source_scrap_id)`。
- 增加普通索引 `INDEX (source_scrap_id)`。
- 不增加独立报废补料关联表。
- 每条补料需求仍直接通过 `source_scrap_id` 关联来源报废。
- 补料幂等由 `production_item_demand.idempotency_key` 保证。

#### 条件规则

- `demand_type = 0` 的正常需求不得关联 `source_scrap_id` 或 `parent_demand_id`。
- `demand_type = 2` 的报废补料必须填写 `parent_demand_id`、`source_scrap_id` 和 `product_material_id`。
- 报废记录、原始需求和补料需求必须属于同一个生产批次。
- `product_material_id` 对应的投入对象必须与补料需求 `item_id` 一致。
- 原始需求的 `need_number` 不得因为报废补料被修改。
- 只有已确认、未取消的报废记录能够生成补料需求。

#### 测试要求

- 一条已确认报废可以生成多条不同 BOM 物料补料需求。
- 跨生产批次的原需求、报废和补料组合必须被拒绝。
- 已取消或待确认报废不能生成补料需求。

### 2.7 分配并发超额预留

#### 现状与问题

`v_item_batch_available_to_allocate` 只能提供查询结果。两个并发事务可能同时读取相同可分配数量并分别成功插入分配，造成总预留超过可用库存。

#### 最终决策

库存正确性使用数据库事务和 `item_batch` 行锁保证，Redis 不作为必要条件或库存事实来源。

#### 应用事务规则

分配事务必须按以下顺序执行：

1. 锁定目标 `item_batch` 行；多批次操作按稳定的批次 ID 升序加锁，避免死锁。
2. 在锁内从 `inventory_transaction` 重新汇总账面可用库存。
3. 在锁内汇总有效、未释放、未取消的生产分配占用。
4. 计算最新可分配数量并校验本次分配。
5. 写入 `production_item_allocation`。
6. 更新必要的业务状态并写操作日志。
7. 提交事务后再向调用方返回成功。

单库存批次的行锁查询应使用等价于以下语句的 Repository 实现：

```sql
SELECT id
FROM item_batch
WHERE id = :batch_id
FOR UPDATE;
```

多库存批次必须先对 ID 排序，再在同一事务中按排序结果逐行加锁；不得先读取汇总视图、释放事务后再写入分配。

出库、退料、库存侧报废和盘点确认涉及同一库存批次时，也必须遵循相同批次锁顺序。

#### 测试要求

- 两个并发分配请求不能使有效预留总量超过可分配库存。
- 两个并发出库请求不能使任何库存状态汇总为负数。
- 多库存批次分配按照固定锁顺序执行，不产生可稳定复现的死锁。
- Redis 关闭时库存并发测试仍然正确。

### 2.8 分配、出库、退料和报废链路一致性

#### 现状与问题

现有设计中 `allocation_id`、`demand_id`、`production_batch_id`、`item_id` 和 `batch_id` 分别引用合法记录时，仍可能组合成不属于同一分配链路的数据。例如出库明细引用分配 A，却填写分配 B 的库存批次。

#### 数据库修改

在 `production_item_allocation` 建立组合候选键：

```text
UNIQUE (
  id,
  demand_id,
  production_batch_id,
  item_id,
  batch_id
)
```

`outbound_detail` 和 `return_detail` 应以相同字段顺序建立完整组合外键，引用对应 allocation 组合候选键。已有能保证主单与生产批次一致的组合外键继续保留。

#### 报废场景规则

- `WAREHOUSE_ALLOCATED`：必须关联有效 allocation、需求、生产批次、物料和尚有可用预留的库存批次；确认后生成负数报废库存流水。
- `RETURN_AFTER_OUTBOUND`：必须通过 `return_detail_id` 精确追溯原 allocation 和已确认退料；报废数量不得超过该退料明细已退回且尚未处置的数量。
- `PRODUCTION_CONSUMED`：物料已在领料时扣减库存，报废确认不得再次生成负库存流水，但必须保留生产批次、需求、分配和原库存批次追溯。
- `IN_STOCK`：不要求生产需求或 allocation，只要求库存对象和库存批次一致，确认后生成负数报废库存流水。

无法通过普通 `CHECK` 或外键表达的跨行数量限制，由应用在批次行锁事务内校验。

#### 测试要求

- 出库或退料使用其他 allocation 的 `batch_id` 时数据库或应用必须拒绝。
- 生产消耗报废不会重复扣减库存。
- 库存内报废不需要虚构生产批次或需求。
- 报废、退料和出库累计数量不得超过对应可处理数量。

### 2.9 库存状态转换

#### 现状与语义

`inventory_transaction.stock_status` 用于区分可用、待检、冻结和不良库存数量。“待检”仍然是库存数量的状态维度，不承载检验结论、检验人员或检验报告。

#### 最终决策

不新增库存状态转换主表或明细表。使用同一事务内的双库存流水表达状态转换，并通过分组键证明两条流水属于同一业务动作。

#### 数据库修改

给 `inventory_transaction` 增加：

```text
transaction_group_key VARCHAR(150) NULL
```

`reference_type` 增加：

```text
INSPECTION_RECORD
```

质量放行产生：

```text
待检库存：状态转出，负数
可用库存：状态转入，正数
```

两条流水必须：

- 使用相同 `transaction_group_key`；
- 使用不同且分别唯一的 `idempotency_key`；
- 具有相同 `item_id`、`batch_id`、单位和数量绝对值；
- 一正一负并在同一数据库事务写入；
- 使用 `reference_type = INSPECTION_RECORD`，`reference_detail_id` 指向触发放行的检验记录。

质量结论、检验人员和报告仍只保存在 `inspection_records`。

#### 测试要求

- 质量放行前后该库存批次总数量不变。
- 待检数量按放行数量减少，可用数量按相同数量增加。
- 任一流水写入失败时整个状态转换回滚。
- 重复放行命令不会重复生成状态转换流水。
- 部分放行不得超过当前待检库存和检验可放行数量。

### 2.10 生产批次最终完工事实

#### 现状与问题

`production_batches` 只有计划数量和状态，没有最终完成数量、最终合格数量及完工确认信息，无法可靠汇总工单完成数量。生产入库是独立业务动作，不能反推生产完工数量。

#### 数据库修改

给 `production_batches` 增加：

```text
completed_quantity DECIMAL(12,4) NOT NULL DEFAULT 0
qualified_quantity DECIMAL(12,4) NOT NULL DEFAULT 0
completed_at       DATETIME NULL
completed_by       BIGINT UNSIGNED NULL
```

约束：

```text
completed_quantity >= 0
qualified_quantity >= 0
qualified_quantity <= completed_quantity
completed_by -> users.id
```

#### 应用事务规则

- 批次完工前校验所有 `need_record_snapshot = 1` 的工序已完成。
- 批次完工前校验所有 `need_inspection_snapshot = 1` 的工序存在有效检验结论。
- 批次完工前校验不存在未关闭返工。
- 使用 `production_batches.version` 做乐观锁，防止重复或并发完工确认。
- 在同一事务写入最终数量、完工时间、完工人、`completed` 状态和操作日志。
- 批次完工不自动创建入库单、库存批次或库存流水。
- 工单完成数量从未取消且已完成的生产批次 `qualified_quantity` 汇总；具体完成阈值和关闭规则在 `docs/new.md` 修订时统一定义。

#### 测试要求

- 合格数量大于完成数量时拒绝完工。
- 必报工、必检验或返工条件未满足时拒绝完工。
- 同一个 `version` 的并发完工确认只有一个成功。
- 批次完工后库存数量不自动变化。

### 2.11 返工复检链路

#### 现状与问题

`rework_records.source_inspection_id` 可以指向触发返工的原检验，但返工完成后的复检不能指回具体返工记录，无法清晰还原“原检验 -> 返工 -> 一次或多次复检”链路。

#### 数据库修改

给 `inspection_records` 增加：

```text
source_rework_id BIGINT UNSIGNED NULL
```

外键 `source_rework_id -> rework_records.id` 在两张表创建后通过后续迁移追加，不修改已执行迁移。

#### 应用事务规则

- 初检和与返工无关的检验必须使用空 `source_rework_id`。
- 返工后的复检必须填写对应返工记录 ID。
- 一个返工允许多次复检，因此 `source_rework_id` 不建立唯一约束。
- 应用必须校验复检、返工来源检验属于同一生产批次；工序级返工还必须属于同一批次工序。
- 原检验和历史复检不得被覆盖或物理删除。

#### 测试要求

- 一个返工可以创建多次复检并完整追溯。
- 不能把其他生产批次的返工记录关联到当前复检。
- 删除或取消返工不得破坏已有检验历史。

### 2.12 退料与报废语义

#### 现状与问题

`return_detail.return_stock_status` 当前允许“报废”，但报废是库存处置动作，不是可继续持有数量的库存状态。把二者混用会导致退料入库和报废出库事实不清晰。

#### 数据库修改

`return_stock_status` 只允许：

```text
可用 / 待检 / 冻结 / 不良
```

从检查约束中删除“报废”。

#### 业务流程

不能继续使用的退料按以下顺序处理：

1. `return_order` 和 `return_detail` 记录退料，退回状态使用“不良”。
2. 退料确认生成正数“不良”退料入库流水。
3. 创建并确认 `item_scrap`，场景使用 `RETURN_AFTER_OUTBOUND`。
4. 报废确认生成负数“不良”报废出库流水。

`return_detail` 只记录退料事实，`item_scrap` 只记录报废事实，两者不得合并或互相替代。

#### 测试要求

- 退料接口不能提交“报废”库存状态。
- 退回不良后报废，库存净变化与实际物料回仓再处置一致。
- 报废失败不会撤销已经真实发生且已确认的退料；应保留不良库存等待后续处理。

### 2.13 多态库存流水来源

#### 现状与限制

`inventory_transaction.reference_type + reference_detail_id` 可能指向入库明细、出库明细、退料明细、报废记录、盘点明细或检验记录。一个普通外键不能根据类型动态引用多张表。

#### 最终决策

保留多态引用，不为此额外拆表。该关系明确标注为受 application 事务约束的引用，是数据库普通外键无法表达的例外。

#### 应用事务规则

- 根据 `reference_type` 校验 `reference_detail_id` 指向的记录存在且类型匹配。
- 校验来源记录与库存流水的 `item_id`、`batch_id`、单位、数量方向和业务状态一致。
- 业务明细、库存流水和操作日志必须在同一事务写入。
- 每条库存流水必须具有唯一 `idempotency_key`。
- 已确认库存流水不可更新或删除，错误只能通过数量相反、库存状态相同的冲销流水修正。
- 冲销流水必须填写 `reversal_of_transaction_id` 并保留原业务引用。
- 一期只支持一次整笔全额冲销，不支持部分冲销；该字段指向 MES 原库存流水，不指向财务报销单或财务凭证。

#### 契约测试要求

每种 `reference_type` 至少覆盖：

- 正确来源可以生成预期方向和状态的库存流水；
- 不存在或类型错误的来源被拒绝；
- 来源对象或批次不一致时被拒绝；
- 相同幂等键重试不会重复记账；
- 冲销后原流水仍保留且汇总数量正确。

### 2.14 软删除编码唯一性

#### 现状与问题

`UNIQUE (业务编码, is_deleted)` 在布尔软删除模型下只能容纳一条相同编码的已删除记录。相同编码第一次删除、重建后，第二次删除会与第一条已删除记录发生唯一键冲突。

#### 最终决策

MES 主数据业务编码永久不复用。删除后如需再次使用，应恢复原记录，不创建同编码的新记录。

#### 数据库修改范围

- 产品编码使用永久唯一约束，不包含 `is_deleted`。
- 产品分类编码使用永久唯一约束，不包含 `is_deleted`。
- 工序编码使用永久唯一约束，不包含 `is_deleted`。
- 工艺路线的产品、路线编码和版本组合永久唯一，不包含 `is_deleted`。
- BOM 等没有独立业务编码的关联唯一约束，在修订 `docs/new.md` 时按历史追溯要求逐项确认，不能机械套用编码规则。
- RBAC、认证和操作日志已经迁移，本次不修改对应迁移和约束。

#### 应用规则与测试

- 创建已存在但已软删除的编码时，返回“编码已存在，可恢复原记录”，不得静默创建新行。
- 恢复记录时校验其关联主数据仍有效，并写操作日志。
- 同一编码经历多次停用、启用或恢复时，不产生重复主数据 ID。
