# 当前项目审计（2026-07-20）

## 结论

现有项目已经具备 pnpm workspace、多个应用、共享包、NestJS 顶层模块、RBAC、事务封装和操作日志雏形，因此不是“从零开始”。主要风险是：目录存在，但架构边界、质量门禁、数据演进和运行保障没有形成闭环。

## 已验证的问题

### P0：交付链路不可作为发布门禁

- `pnpm build` 失败：管理端存在查询参数类型不一致、Vue Ref 使用错误等 TypeScript 问题。
- `pnpm lint:eslint` 失败：8 个 error、1 个 warning。
- `pnpm test` 返回成功，但 9 个工作区均无测试文件，并通过 `--passWithNoTests` 放行。
- 仓库没有 CI/CD 工作流、Dockerfile 或 Compose 部署定义。

### P0：数据库版本不可重复构建

- SQL 文件兼有全量备份、破坏性重建脚本和增量 ALTER，缺少统一迁移版本表、不可变规则、校验和、状态检查与失败恢复流程。
- `newSqlDesign.md` 已演进为统一库存对象方案，但历史 SQL 和旧文档仍保留多套命名，容易让代码、环境和文档继续漂移。
- 暂无“从空库构建”与“从上一生产版本升级”两条 CI 验证路径，也没有 schema drift 检测。

### P1：后端只是顶层模块化，模块内部仍偏事务脚本

- 大量 Controller 直接接收共享 TypeScript interface，没有运行时 DTO 校验；未发现全局 ValidationPipe。
- Repository 同时承担 SQL、聚合、状态流转和映射，最大文件超过 1100 行，应用服务层缺失或过薄。
- Product / Production / Warehouse 模块注册大量 Controller 和 Repository，但缺少明确的 domain、application、infrastructure 边界。
- 权限 Guard 只在声明了权限元数据时生效；应改为默认鉴权、显式 Public，避免新接口忘记加装饰器。

### P1：安全与运行保障不足

- JWT secret 存在开发默认值，配置未用统一 schema 在启动时完整校验。
- 未发现 Helmet、全局异常规范、请求关联 ID、结构化日志、限流、优雅停机、就绪/存活探针和依赖健康检查。
- `/health` 是静态响应，不能反映 MySQL、Redis 或存储是否可用。
- 上传在 Controller 中直接把内存文件写入本地目录，未见统一大小限制、MIME/扩展名策略、病毒扫描钩子、对象存储抽象或孤儿文件清理。
- 审计日志采用异步 fire-and-forget 写入；进程退出或写入失败时可能丢日志，且缺少敏感字段脱敏策略。

### P1：前端维护成本快速上升

- 只有 25 个 Vue 文件，但多个页面在 600～950 行，列表、查询、表单、弹窗、字典和业务状态集中在单文件。
- API contract 只有编译期 interface，没有运行时 schema，前后端字段漂移已直接表现为构建失败。
- 缺少统一的页面壳、查询栏、数据表、分页、Modal 表单、状态标签、权限按钮和错误态组件约束。
- 工作站端与管理端的共享边界尚未通过可测试的 UI / composable 包稳定下来。

### P2：可观测性、恢复与治理缺失

- 没有指标、追踪、慢 SQL、错误聚合、告警规则和容量基线。
- 没有备份恢复演练、RPO/RTO、数据库迁移回退和文件存储恢复 runbook。
- 没有依赖更新、漏洞扫描、镜像扫描、SBOM、秘密扫描和许可证检查。
- 没有 ADR、模块负责人、发布记录、环境晋级规则和变更审批边界。
- API 没有 OpenAPI 契约与兼容性检查，分页、错误码、幂等键和并发冲突处理未统一。

## 规模判断

约 50 人使用时，性能风险首先来自慢 SQL、缺少索引、超大查询、连接池和错误的事务边界，不是 Redis 缺席本身。建议默认 MySQL + 单实例 API，保留 Redis 适配器；只有出现多实例、热点缓存、跨实例锁、队列或限流一致性需求时再启用 Redis。
