# 生产业务完整工作流

本流程不包含管理员、角色和权限控制，仅描述产品、工艺、生产、质量和库存业务。

文中数据库字段值统一使用小写英文 `snake_case` 稳定代码，中文只用于业务说明和前端标签。数据库继续使用 `VARCHAR + CHECK`，不使用 MySQL `ENUM`；状态值域由数据库约束，合法状态转换由后端领域层控制。

## 一、基础资料准备

### 1. 配置产品分类

进入产品分类管理，配置物料、半成品和成品分类。

分类信息保存至 `product_categories`，通过 `item_kind` 区分：

- `material`：物料；
- `semi_finished`：半成品；
- `finished_product`：成品。

物料、半成品和成品统一保存在 `products` 表中，通过 `products.category_id` 关联分类。这里不是单独的“产品与物料映射表”，而是产品表直接关联分类表。

### 2. 添加物料或产品资料

在 `products` 中添加库存对象，并根据所选分类确定其属于物料、半成品还是成品。

本次添加为物料时，只维护物料编码、名称、单位、获取方式和规格等基础资料，不配置 BOM 和工艺路线。产品、物料和半成品统一使用 `item_code` 作为唯一业务编码，不单独维护型号字段。

对于自产半成品或成品，可以继续配置：

- 产品物料清单；
- 产品工艺路线；
- 默认工艺路线。

系统应根据分类和获取方式进行业务校验，例如采购物料原则上不能配置生产工艺路线，自产半成品和自产成品可以配置 BOM 和工艺路线。

### 3. 物料入库

物料每次入库都应按库存批次管理。

入库时：

1. 在 `item_batch` 中建立物料库存批次；
2. 创建 `inbound_order` 入库单；
3. 创建 `inbound_detail` 入库明细；
4. 生成正数的 `inventory_transaction` 库存流水。

库存现存数量以 `inventory_transaction` 的汇总结果为准，不直接在物料或批次表中维护可修改的库存余额。

### 4. 添加标准工序

在 `process_steps` 中维护标准工序，例如装配、焊接、调试、检验和包装。

工序可以关联默认 SOP 文件。文件信息统一保存在 `technical_files` 中，工序通过 `default_sop_file_id` 关联默认 SOP。

### 5. 配置工艺路线

创建 `process_routes` 工艺路线，并通过 `process_route_steps` 按顺序组合多个工序。

每个路线步骤可以配置：

- 工序顺序；
- 默认负责人；
- 本步骤使用的 SOP 文件；
- 是否需要报工；
- 是否需要检验。

工艺路线必须绑定具体产品。路线启用后不得直接修改原版本，如需调整工序顺序或 SOP，应创建新的路线版本。

### 6. 配置成品或半成品

再次添加产品，本次选择成品或自产半成品，并完成以下配置：

1. 在 `product_materials` 中配置产品 BOM；
2. 为每种投入物料配置单位用量和用量单位；
3. 选择该产品使用的工艺路线；
4. 将默认路线保存至 `products.default_route_id`。

产品 BOM 是后续生成生产物料需求的唯一数据来源。修改 BOM 不得影响已经生成的历史生产需求。

## 二、生产计划与派工

### 7. 创建生产工单

创建 `work_orders` 工单，选择需要生产的产品，并填写工单计划生产数量。

工单下达时，应保存产品编码、产品名称和单位快照，后续修改产品资料不得影响历史工单。

工单初始状态为 `draft`，确认下达后变为 `released`。

一个工单表示整体生产计划，可以拆分成多个生产批次执行。

工单可以关联一个可空的外部订单号，支持同一个外部订单拆分为多个生产工单。外部订单号不作为 MES 工单自身的唯一标识；`work_order_no` 仍保持唯一。

### 8. 创建生产任务

系统中的“生产任务”对应 `production_batches` 生产批次，不需要再建立一张独立任务表。

创建生产批次时：

1. 绑定一个工单；
2. 自动带出工单产品；
3. 填写本批次计划生产数量；
4. 选择该产品的工艺路线（未指定时读取产品默认路线，指定路线时允许使用同产品的非默认路线）；
5. 指定本批次总体负责人；
6. 冻结路线编号和路线版本快照。

一个工单可以拆成多个生产批次，不要求一个批次完成工单的全部生产数量。

创建批次时，后端必须执行以下校验（同一应用事务内）：

- 该工单下未取消生产批次的计划数量合计不得超过工单计划数量。
- 所选路线必须属于工单产品，且路线状态为 `enabled`。
- 批次工序由后端按路线步骤顺序自动生成，不接受前端提交任意 `route_step_id` 集合。
- 路线中的必须报工和必须检验标志同时冻结为快照。

### 9. 生成工序执行记录并完成派工

生产批次创建后，系统根据所选工艺路线自动生成 `batch_step_records`。

每个路线步骤生成一条工序执行记录，并复制：

- 工序顺序；
- 工序编码和名称；
- SOP 文件信息；
- 默认负责人；
- 报工单位；
- 必须报工标志快照；
- 必须检验标志快照。

路线中的默认负责人只作为派工表单的默认值。生产批次创建时可以重新指定实际负责人，最终实际负责人保存在 `batch_step_records.responsible_user_id` 中。

后续调整路线、工序默认负责人或 SOP 文件，不得回写已经生成的生产批次工序记录。

当前数据库没有生产批次级的统一“工艺参考文件”字段，因此工艺文件应按工序保存在 `batch_step_records` 的 SOP 快照字段中。

## 三、物料需求、预留与领料

### 10. 生成物料需求

生产批次创建后，可以手动点击或由系统自动生成物料需求。

系统根据以下数据计算需求：

```text
需求数量 = BOM 单位用量 × 生产批次计划数量
```

每种物料生成一条 `production_item_demand`，并保存：

- BOM 明细 ID；
- 物料 ID；
- 单位用量快照；
- 单位快照；
- 批次追溯要求快照；
- 生产批次计划数量快照；
- 最终需求数量；
- 幂等键，防止重复点击或重试重复创建需求。

幂等键生成规则：

- 正常需求：`NORMAL:{production_batch_id}:{product_material_id}`
- 报废补料：`SCRAP:{source_scrap_id}:{product_material_id}`
- 人工追加：`ADDITIONAL:{production_batch_id}:{business_action_no}:{product_material_id}`

同一个幂等键重复提交时返回既有结果，不再插入新需求。

需求生成后作为历史事实保存。后续修改 BOM 或生产批次计划数量，不得自动覆盖原有需求；如需调整，应取消原需求或新增追加需求。

此时生产批次可进入 `material_pending` 状态。

### 11. 分配物料并形成库存预留

仓库根据物料需求进行物料分配，分配结果保存至 `production_item_allocation`。

同一种物料可以从多个 `item_batch` 库存批次中分配，例如：

```text
需求 100 个
库存批次 A 分配 60 个
库存批次 B 分配 40 个
```

每条分配记录必须关联：

- 生产物料需求；
- 生产批次；
- 物料；
- 库存批次；
- 分配数量。

创建物料分配记录即表示完成库存预留。物料需求本身只表示计划，不会占用库存。

系统分配时必须查询可分配库存，不能只查询账面库存：

```text
可分配库存 = 账面可用库存 - 其他生产批次已预留但未释放的数量
```

分配必须使用 `item_batch` 行锁防止并发超额预留。分配事务执行顺序：

1. 锁定目标 `item_batch` 行（多批次按 ID 升序加锁）；
2. 在锁内重新汇总账面可用库存；
3. 在锁内汇总有效、未释放、未取消的分配占用；
4. 计算最新可分配数量并校验；
5. 写入 `production_item_allocation`；
6. 更新业务状态并写操作日志；
7. 提交事务后再返回成功。

物料全部分配完成后，生产批次进入 `material_assigned` 状态。

### 12. 生产领料出库

物料分配完成后，仓库进行实际领料出库。

领料时：

1. 创建 `outbound_order`；
2. 按分配记录创建 `outbound_detail`；
3. 为每条出库明细生成负数的 `inventory_transaction`；
4. 库存流水关联具体的 `outbound_detail.id`；
5. 生产批次进入 `material_outbound` 状态。

物料分配只表示预留，不会减少账面库存；只有领料出库才会实际扣减库存。

因此，标准流程应为：

```text
生成需求 → 分配预留 → 领料出库 → 生产开工
```

不能把“物料已分配”直接等同于“物料已出库”。

## 四、生产执行与质量闭环

### 13. 生产开工

领料完成后，生产批次进入 `doing` 状态。

每道工序开始时，更新对应 `batch_step_records`：

- 状态修改为 `doing`；
- 记录 `started_at`；
- 确认实际负责人；
- 展示该工序冻结的 SOP 文件。

系统原则上按照 `step_order_snapshot` 顺序执行工序。前一道必报工工序未完成时，后一道工序是否允许开工，由后端业务规则校验。

### 14. 工序报工

每道工序完成后，在对应的 `batch_step_records` 中进行报工，填写：

- 完成数量；
- 合格数量；
- 异常数量；
- 返工数量；
- 完工时间；
- 报工备注。

正常完成时，工序状态变为 `completed`。

出现异常但尚未完成处理时，可以将状态设置为 `abnormal`。

数量必须满足：

```text
合格数量 + 异常数量 ≤ 完成数量
```

工序完工时必须已经存在开工时间，并且完工时间不得早于开工时间。

### 15. 工序检验与返工

如果工艺路线步骤配置了需要检验，则工序报工后创建 `inspection_records`。

检验记录可以关联：

- 生产批次；
- 具体的工序报工记录；
- 检验数量；
- 合格数量；
- 不合格数量；
- 检验结论；
- 检验报告文件。

检验通过后，工序可以继续向下一步骤流转。

检验不合格但允许返工时，创建 `rework_records`，记录返工数量、返工原因、返工措施、负责人和返工结果。

返工完成后可再次创建检验记录进行复检，复检记录通过 `source_rework_id` 指向具体返工记录，形成"原检验 → 返工 → 复检"可追溯链路：

- 初检和与返工无关的检验必须使用空 `source_rework_id`。
- 一个返工允许多次复检。
- 原检验和历史复检不得被覆盖或物理删除。
- 应用必须校验复检、返工来源检验属于同一生产批次。

### 16. 损坏、异常和报废补料

生产过程中出现无法维修的损坏或不合格时，应进行以下处理：

1. 在当前工序的 `batch_step_records` 中填写异常数量，并标记工序异常；
2. 创建 `item_scrap` 报废记录；
3. 记录报废对象、报废数量、报废原因和报废场景；
4. 报废确认后，根据实际情况决定是否生成补料需求。

如果物料已经领出并在生产过程中损坏，报废场景使用 `production_consumed`。由于领料时库存已经扣减，此时不得再次生成库存扣减流水。

如果仓库内已分配但尚未出库的物料发生报废，则应记录库存报废，并生成对应的负数库存流水。

需要补料时，不得直接增加原始需求的 `need_number`，应新增 `production_item_demand`：

- `demand_type = 2`；
- `parent_demand_id = 原始需求 ID`；
- `source_scrap_id = 报废记录 ID`；
- `need_number = 本次补料数量`。

一条已确认报废记录可以针对多种 BOM 物料分别生成补料需求。每种物料生成一条独立补料需求记录，均关联同一 `source_scrap_id`。

补料需求生成后，重新执行：

```text
补料需求 → 物料分配 → 领料出库
```

条件规则：

- `demand_type = 0` 的正常需求不得关联 `source_scrap_id` 或 `parent_demand_id`。
- `demand_type = 2` 的报废补料必须填写 `parent_demand_id`、`source_scrap_id` 和 `product_material_id`。
- 报废记录、原始需求和补料需求必须属于同一个生产批次。
- 只有已确认、未取消的报废记录能够生成补料需求。

### 17. 生产退料

生产过程中或生产结束后，如存在未使用物料，可以办理退料。

退料时：

1. 创建 `return_order`；
2. 创建 `return_detail`；
3. 记录原需求、原分配记录、物料批次和退料数量；
4. 生成正数的 `inventory_transaction` 退料入库流水。

退料时需要明确是否释放给公共库存：

- `release_after_return = 0`：仍属于原生产批次，后续可以再次领用；
- `release_after_return = 1`：释放给公共库存，其他生产批次可以重新分配。

退料退回的库存状态只允许 `available`（可用）、`pending_inspection`（待检）、`frozen`（冻结）、`defective`（不良），不含“报废”。不能继续使用的退料按以下顺序处理：

1. `return_detail` 记录退料，退回状态使用 `defective`（不良）。
2. 退料确认生成 `stock_status = defective` 的正数退料入库流水。
3. 创建并确认 `item_scrap`，场景使用 `return_after_outbound`，并通过 `return_detail_id` 精确关联来源退料明细。
4. 报废确认生成 `stock_status = defective` 的负数报废出库流水。

`return_detail` 只记录退料事实，`item_scrap` 只记录报废事实，两者不得合并或互相替代。

## 五、完工与入库

### 18. 生产批次完工

生产批次完工时，填写最终完成数量和最终合格数量，确认后保存：

- `completed_quantity` — 本批次最终完成数量
- `qualified_quantity` — 本批次最终合格数量（≤ completed_quantity）
- `completed_at` — 完工确认时间
- `completed_by` — 完工确认人

使用 `production_batches.version` 乐观锁防止重复或并发完工确认。

当以下条件满足后，可以将生产批次状态修改为 `completed`：

- 所有必须报工（`need_record_snapshot = 1`）的工序均已完成；
- 所有必须检验（`need_inspection_snapshot = 1`）的工序已存在有效检验结论；
- 不存在未关闭的返工；
- 最终完成数量和合格数量已经确认，且 `qualified_quantity <= completed_quantity`。

生产批次完成不等于成品已经入库。生产完成和库存入库是两个独立业务动作。

批次完工不自动创建入库单、库存批次或库存流水。

当工单下所有有效生产批次均已完成，且完成数量达到工单计划数量后，工单可以修改为 `completed`；后续业务全部结束后可修改为 `closed`。

### 19. 半成品或成品入库

生产完成后，对合格半成品或成品办理生产入库。

入库时：

1. 在 `item_batch` 中创建新的库存批次；
2. 设置 `source_type = self_made`；
3. 关联来源工单和来源生产批次；
4. 创建 `inbound_order`；
5. 创建 `inbound_detail`；
6. 生成正数的 `inventory_transaction`；
7. 创建 `finished_flow_records`，记录成品或半成品入库里程碑。

一个生产批次可以生成多个库存批次，例如：

- 不同生产日期的库存批次；
- 不同质量状态的库存批次；
- 半成品库存批次；
- 成品库存批次。

如果产品入库后仍需等待最终检验，可以先按 `pending_inspection`（待检）状态入库。

质量放行时，不修改原库存流水，而是在同一事务内创建两条库存状态转换流水：

1. `stock_status = pending_inspection`，数量为负数的状态转出流水；
2. `stock_status = available`，数量相同为正数的状态转入流水。

两条流水共享相同 `transaction_group_key`，使用不同且分别唯一的 `idempotency_key`，`reference_type` 为 `inspection_record`，指向触发放行的检验记录。

质量结论、检验人员和报告仍只保存在 `inspection_records`。库存流水只记录库存数量和状态维度。

库存数量只能由 `inventory_transaction` 汇总得出，`finished_flow_records` 只记录流转里程碑，不直接改变库存数量。

## 六、库存盘点与修正

### 20. 创建库存盘点

创建 `stock_check_order` 盘点单，并选择需要盘点的库存对象、库存批次和库存状态。

系统生成 `stock_check_detail`，记录盘点时的：

- 物料或产品；
- 库存批次；
- 库存状态；
- 系统账面数量；
- 实际盘点数量；
- 差异数量；
- 盘盈、盘亏或一致结果。

### 21. 确认盘点差异

盘点确认后，根据差异生成 `inventory_transaction`：

- 盘盈生成正数的“盘点调整”流水；
- 盘亏生成负数的“盘点调整”流水；
- 无差异时不生成库存流水。

生成调整流水后，将 `stock_check_detail.adjusted` 标记为 `1`，避免重复调整。

盘点修正不得直接修改库存余额，也不得修改或删除原库存流水。错误的库存流水必须通过数量相反的冲销流水修正。一期只支持一次整笔全额冲销；该冲销是 MES 库存流水纠错，与财务报销单或财务凭证无关。

## 七、核心状态流转

生产工单状态：

```text
draft → released → doing → completed → closed
```

生产批次状态：

```text
pending
→ material_pending
→ material_assigned
→ material_outbound
→ doing
→ completed
```

工序执行状态：

```text
pending → assigned → doing → completed
```

异常情况下可进入：

```text
abnormal
```

取消的工单、生产批次、入库单或出库单通过取消状态表达，不物理删除已经产生的业务事实。已经确认的库存变动必须通过冲销流水处理。

## 八、设计边界说明

以下为数据库设计中明确的业务边界：

1. 系统不引入项目模型，不提供项目级数据隔离。“进入生产管理”不表示系统已具备多项目权限或库存隔离能力。
2. 工单关联的外部订单号 `external_order_no` 只用于查询，不保证唯一。同一个外部订单可拆分为多个工单。
3. 路线与产品归属由后端应用事务校验，不依赖数据库组合外键。批次可使用同产品的非默认路线。
4. 工序规则（必须报工、必须检验）在批次创建时冻结为快照；后续路线调整不影响历史批次。
5. 生产需求由幂等键保证不重复生成，正常需求、报废补料和人工追加使用不同键前缀。
6. 报废补料支持一条报废记录生成多种 BOM 物料的需求，不再受 `source_scrap_id` 唯一约束限制。
7. 分配并发安全由 `item_batch` 行锁保证，不依赖 Redis。
8. 库存状态转换使用同事务内一正一负双流水，`transaction_group_key` 保证两条流水属于同一动作。
9. 批次完工保存最终完成数量和合格数量，完工不自动创建入库单或库存流水。
10. 复检通过 `source_rework_id` 追溯具体返工记录，一个返工可多次复检。
11. 退料库存状态不含“报废”，不能继续使用的退料独立走报废流程。
12. MES 主数据编码永久唯一，软删除后通过恢复原记录处理，不创建同编码新记录。

除上述边界外，完整业务主线为：

```text
配置分类
→ 添加物料和产品
→ 物料入库
→ 配置工序、工艺路线和 BOM
→ 创建工单
→ 创建生产批次并生成工序任务
→ 生成物料需求
→ 分配物料并形成预留
→ 领料出库
→ 工序开工和报工
→ 检验、返工、报废及补料
→ 生产完工
→ 半成品或成品入库
→ 退料及库存盘点修正
→ 生产与库存追溯查询
```
